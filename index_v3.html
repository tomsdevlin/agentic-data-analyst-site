AI Analyst

AI-powered data analysis that runs entirely on your own hardware. Your data never leaves your system — no cloud, no external servers, no loss of control.

Why this exists

Ever wanted to take advantage of AI for data analysis, but not comfortable with your data going out to the internet and losing control of it? That is why AI Analyst exists.

Most AI analysis tools require you to send your data to a cloud server, where it passes through external infrastructure, may be logged, and sits outside your control. AI Analyst runs entirely on your own hardware. The language models are local, the computation is local, and your data never travels anywhere.

Beyond privacy, it also solves the translation problem that slows down analytical work: turning a business question into code, running it correctly, and writing up findings that are accurate and qualified. AI Analyst handles that workflow end-to-end — goal statement in, structured written findings out — without requiring any code from you.

The four-stage pipeline

Every analysis runs through the same sequence of stages. Each stage has a distinct role, uses a dedicated language model, and passes a structured result to the next stage. The pipeline is designed so that any stage can fail independently — a planning failure doesn't run the executor, and a synthesis failure surfaces a clear error, not a silent wrong answer.

Stage by stage

Each stage is a separate, independently testable component. Expand any stage to see what it does.

1
Planner
Breaks the goal into a sequence of analytical questions

The planner reads the dataset's column schema and retrieves the most relevant analytical playbooks, then produces a structured plan: a numbered list of plain-English questions that, taken together, address the goal.

Each question is mapped to a dimension of the goal — for example, "trend over time", "breakdown by category", "outlier identification" — and flagged as a text question or a chart query. The planner is required to cover every stated dimension; a warning is raised if any goes unaddressed.

You can also add supplementary context when entering a goal in the app — for example, a known data quality issue, a field to prioritise, or a time period to focus on. This context is passed directly to the planner alongside the goal and shapes the analytical plan from the start.

Takes in
Your goal statement, any supplementary context from the app, the dataset's column schema, and the matching playbooks
Produces
A structured list of questions with goal dimension tags and dependency links
2
Executor
Writes and runs Python for each question, then narrates the result

For each planned question, the executor decides whether it needs to run code or can be answered directly from the schema. For code questions, it writes Python using pandas, runs it in a sandboxed environment, captures the output, then calls a second model to narrate the result in plain English.

Questions that can run independently execute in parallel. Questions that depend on a prior result (e.g. "now break that down by region") run in sequence, with previous results in context. This means a 10-question plan often runs in the time of 3–4 queries.

The narration — not the raw code output — is what flows into subsequent stages and into the conversation history for follow-up questions.

Takes in
The question list, the dataset, and a growing history of prior results
Produces
Per-question: code, raw output, plain-English narration, any charts generated
3
Synthesiser
Turns individual query results into structured findings

The synthesiser reads all query results together and produces a set of findings — each with a title, an interpretation in plain English, a confidence level (High / Medium / Low), a sensitivity indicator, and supporting query references.

It works within constraints: findings must have distinct claims (no two findings may assert the same relationship), magnitude words like "dominates" or "majority" require a percentage to back them up, and any comparison across groups must note whether the groups are similar in size. Findings are only generated for goal dimensions where the executor returned usable results.

The output also includes identified gaps — dimensions the planner intended to cover but where the data wasn't sufficient — and two lists of suggested next steps: those you can run immediately on the current dataset, and those that would need additional data.

Takes in
All query narrations, the goal, playbook guidance, and a dimension coverage map
Produces
Structured findings with confidence levels, gaps identified, and next steps split by what's runnable now vs what needs more data
4
Reviewer & Revision Loop
Checks every finding against the evidence; rewrites problems

Each finding is reviewed independently against the evidence from its cited queries. The reviewer checks for a specific set of problems — false claims, unsupported magnitude words, missing caveats, and known analytical traps — and raises typed flags when it finds them.

Flags that indicate a correctable error trigger a targeted revision: only the affected findings are rewritten, not the whole synthesis. The revision loop runs a maximum of two rounds. If a problem persists after two rounds, it remains flagged in the output with the overall verdict downgraded accordingly.

Number-grounding — checking that a specific figure cited in a finding actually appears in the query evidence — is done deterministically before the language model reviews anything. This prevents the model from raising false alarms on numbers that are clearly present.

Takes in
Each finding paired with its supporting query evidence
Produces
Typed flags per finding, a revision if needed, and an overall verdict (PASS / REVIEW / FAIL)

Built-in quality checks

The reviewer classifies every problem it finds into one of six types. The type determines what happens next: correctable problems trigger a rewrite, advisory problems are flagged without blocking.

✗ False Claim

The finding states a number or direction that contradicts the evidence. The finding is rewritten to reflect what the data actually shows.

✗ Unsupported Magnitude

Words like "dominates" or "the majority" appear without a percentage to back them. The claim is requalified with the actual figures.

✗ Missing Caveat

The conclusion is directionally correct but omits a material qualification — for example, comparing groups without noting very different group sizes.

✗ Analytical Pitfall

A known statistical trap has been triggered — such as comparing rates across groups without accounting for composition differences.

⚠ Data Quality Concern

The data limits confidence in the finding — for example, fewer than 15 records in a comparison group, or more than 10% missing values in a key field. Not correctable; flagged for the reader.

⚠ Scope Gap

A dimension of the original goal was not addressed, typically because the data wasn't sufficient to support it. Flagged so the reader knows what was left out.

Confidence is adjusted automatically after review. Findings that were corrected for a False Claim or Pitfall are capped at Low confidence. Findings with Unsupported Magnitude, Missing Caveat, or Data Quality concerns are capped at Medium. High confidence is reserved for findings that passed review without correction.

Under the hood

Five components shape how the system works on a new dataset: a privacy-first local design, sandboxed code execution, a data dictionary, a set of analytical playbooks, and a set of guardrails that constrain what the language models are allowed to decide for themselves.

Privacy by design

Your data and the AI models never leave your system

Most AI analysis tools work by sending your data to a remote server — the model runs in the cloud, results come back, and your data has left your control. AI Analyst is built the other way round: the language models are downloaded and run on your own hardware using Ollama, an open-source local model runtime. No data is transmitted to an external API at any point.

Everything stays on your machine:

  • Your dataset — never uploaded or transmitted anywhere
  • Your goal text and any context entered in the app — processed only by the local model
  • All AI reasoning and generated code — runs entirely on local hardware
  • Results and findings — generated locally and displayed to you only
Sandboxed execution

Code runs in a restricted environment, not a general Python interpreter

The executor writes Python code and runs it to answer each analytical question. That code runs in a restricted sandbox — not a general Python interpreter with access to your files or network:

  • Data access. Only the specific dataset file is accessible to the executor. It cannot read other files on your system.
  • No network calls. The sandboxed execution context has no outbound network access.
  • Isolated queries. Each query runs independently — code from one question cannot affect another.
  • Blocked patterns. Known unsafe operations — writing files, making system calls, launching subprocesses — are intercepted before execution.
Data dictionary

Helping the system understand your data

To get started, drop your CSV file into the app. You can also provide a plain-English data dictionary alongside it — a text file that describes what each column means, what categorical values represent, and how fields relate to each other. If you don't have one, the system generates one automatically from your column names and data types.

Providing your own dictionary gives the planner richer context: the difference between a raw value and a pre-computed metric, known data quality issues, or which fields are sensitive. This context is passed to the synthesiser, which uses it to qualify findings appropriately.

You can also add free-text context when entering a goal in the app — useful for session-specific constraints, known caveats, or fields to prioritise that aren't part of the standing dictionary. This is passed directly to the planner alongside the goal.

Analytical playbooks

Twelve analytical patterns, already built in

Twelve playbooks covering the most common analytical patterns are already included and ready to use. Each one specifies when it applies, how to structure the analysis, what pitfalls to avoid, and what the output should include. Think of them as the organisation's analytical standards — encoded once and applied consistently.

Trend Analysis Group Comparison Composition / Breakdown Top-N Ranking Distribution Analysis Correlation Investigation Hypothesis Testing Anomaly / Outlier Detection Cohort Analysis Funnel / Conversion Analysis Forecasting / Projection Basket / Co-occurrence Analysis

The planner doesn't apply all twelve to every goal. It retrieves the two or three most relevant playbooks using semantic search against the goal statement, then uses them to shape the analytical plan. A trend question triggers the Trend Analysis playbook; a question about which category performs best triggers Group Comparison and Top-N Ranking. Playbooks can be extended or customised without changing any code.

Model freedom vs guardrails

The language model reasons freely; critical operations are deterministic

The planner, synthesiser, and reviewer express themselves freely in natural language — this is where the analytical reasoning happens. But several operations are handled deterministically to prevent well-known failure modes:

  • Question classification. Before asking the model whether a question needs code or text, pattern-matching rules catch obvious cases — questions that start with "Plot…" or "What is the mean X for each Y?" always go to code. This prevents the model from choosing to answer a calculation question in text from memory.
  • Number grounding. Before the reviewer's language model can raise a False Claim flag, the system checks deterministically whether the cited number actually appears in the query evidence. If it does, the flag is suppressed — the LLM cannot override a passing deterministic check.
  • Code post-processors. Known bad code patterns are rewritten automatically after the executor generates them: ratio-of-means calculations are converted to row-level ratios, year axes are cast to integers, and escaped newline sequences are restored to real line breaks before execution.
  • Bounded revision. The revision loop runs a maximum of two rounds. A merge guard ensures the model can only rewrite findings it was explicitly asked to revise — it cannot silently alter correct findings during a revision pass.

The design principle is: let the model do what language models are good at (reasoning, writing, identifying patterns), and use deterministic rules for anything where a silent wrong answer is worse than an explicit constraint.
