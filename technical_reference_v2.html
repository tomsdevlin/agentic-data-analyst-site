Agentic Data Analyst — Technical Reference

Architecture, function signatures, data structures, and design decisions. For contributors and integrators.

System Design

A local-first multi-stage agentic pipeline for structured data analysis. Takes a natural-language analytical goal and a CSV; returns a structured brief with findings, confidence levels, and caveats — all computed on-device using Ollama-served LLMs. No data leaves the machine.

The system uses four specialised LLM roles (planner, executor, synthesiser, reviewer) chained through typed dataclass contracts. Each stage can fail independently; the pipeline continues where possible and logs everything to JSONL.

Key invariants
Stages communicate only through typed dataclasses — no shared mutable state. Each stage receives a complete snapshot of prior stage output. The pipeline logs a full JSONL record regardless of stage outcome, enabling offline replay and evaluation.

Q&A Tab — Classifier Path

The Q&A tab (single free-form question mode) follows a different path — it does not invoke the planner, synthesiser, or reviewer. Instead, it uses a dedicated classifier model to decide how to answer before calling the executor.

Classifier is Q&A tab only. The pipeline tab (planner → executor → synthesiser → reviewer) bypasses the classifier entirely — the executor always receives expected_action="code" directly. Classifier regressions are only visible via python validation_itunes_v2.py or the Q&A tab, not in CORGI pipeline runs. The classifier model (CLASSIFIER_MODEL) is hardwired at the classify_need_for_code() call site and is not overridable by the executor model selector in the Streamlit sidebar.

Pipeline Flow — Pseudocode

def run_analysis(goal: str, df: DataFrame,
               collections_config: dict, config: PipelineConfig)
               -> PipelineResult:

  # 1. Retrieve analytical playbooks from vector store
  playbooks = retrieve_playbooks(goal, k=config.playbook_k)

  # 2. Planner: decompose goal into ordered queries
  plan = generate_plan(goal, df, playbooks, model=config.planner_model)
  if not plan.success: return PipelineResult(failed_stage="plan")

  # 3. Executor: run queries in dependency waves
  exec_result = execute_plan(plan.plan, df, collections_config,
                             model=config.executor_model,
                             max_workers=config.executor_max_workers)
  # 4. Build coverage map: which goal dimensions were answered
  coverage_map = _build_coverage_map(plan, exec_result)

  # 5. Synthesiser: ground findings in executor evidence
  synthesis = synthesise(goal, exec_result, playbooks, coverage_map,
                         model=config.synthesiser_model)
  if not synthesis.success: return PipelineResult(failed_stage="synthesise")

  # 6. Reviewer + revision loop (max 2 rounds)
  review = review(synthesis, exec_result, playbooks, model=config.reviewer_model)
  for round in range(config.max_revision_rounds):
    if not _has_correctable_flags(review): break
    synthesis = _targeted_resynthesize(synthesis, review, exec_result)
    review    = review(synthesis, exec_result, playbooks)

  # 7. Confidence post-adjustment: cap based on unresolved flags
  synthesis = _apply_confidence_adjustments(synthesis, review)

  # 8. Overall status + vacuous-PASS guard
  status = review.overall_assessment   # "PASS" | "REVIEW" | "FAIL"
  if status == "PASS" and exec_result.pass_count < 2: status = "REVIEW"

  _log_pipeline_result(result)   # JSONL → pipeline_logs/
  return PipelineResult(overall_status=status, ...)

executor_loop.py Executor Loop

Runs the analysis plan against the dataset. Groups queries into dependency waves using depends_on fields; within each wave, queries run concurrently via ThreadPoolExecutor. Conversation history is updated between waves so dependent queries see prior results.

execute_plan(
  plan: list[dict],
  df: pd.DataFrame,
  collections_config: dict | None,
  model_name: str = MODEL_NAME,
  max_workers: int = EXECUTOR_MAX_WORKERS
) → ExecutionResult

Wave Scheduler

Queries are sorted into execution waves: a query enters wave N+1 once all its depends_on indices have completed in earlier waves.

def _build_waves(plan: list[dict]) -> list[list[int]]:
  assigned = set()
  waves = []
  while len(assigned) < len(plan):
    wave = [
      i for i, step in enumerate(plan)
      if i not in assigned
         and all(d in assigned for d in normalise_deps(step["depends_on"]))
    ]
    if not wave: wave = [first_unassigned()]  # cycle-break: append one
    waves.append(wave); assigned.update(wave)
  return waves

After each wave completes, results are appended to conversation_history as narrated summaries (not raw code). The next wave's queries receive this updated history via ask_agent_xml().

No retry on code execution failure. Deliberate design decision — with local models, same-model retries rarely produce different code. A fallback model (EXECUTOR_FALLBACK_MODEL) is a genuinely different attempt. SystemExit from generated code is caught explicitly before the generic Exception handler so sys.exit() calls become a failed query result rather than a pipeline crash.

Single-Query Execution: _execute_one()

The unit of work submitted to each ThreadPoolExecutor future. Takes a plan step and returns a fully populated result dict. Catches BaseException (not just Exception) so any exception — including SystemExit, KeyboardInterrupt, and GeneratorExit — becomes a failed query result rather than escaping to the wave loop.

def _execute_one(i, step, df, model_name, collections_config,
                conversation_history, charts_dir, timestamp) -> dict:

  try:
    result = ask_agent_xml(
      query=step["question"], df=df, model_name=model_name,
      collections=collections_config,
      conversation_history=list(conversation_history),  # snapshot — not live ref
    )
  except BaseException as e:
    return {"success": False, "error": str(e), ...}

  # action == "text":  result_text = content; success = bool(content)
  # action == "code":  run exec_result["ok"]; save chart if figure_bytes present
  # action == "error": success = False
  # frames: list of DataFrames captured by _sandboxed_print, propagated from run_sandboxed()

  result_dict["_raw_result"] = result  # stripped after wave completes
  return result_dict

synthesiser.py Synthesiser

Converts executor query results into a structured analytical brief. Each finding must cite the executor query indices that support it. A coverage map injected into the prompt prevents findings from being written for failed or missing dimensions.

) → SynthesisResult

Post-processing: calibrate_language()

calibrate_language(synthesis: SynthesisResult, exec_result: ExecutionResult) → SynthesisResult

Deterministic post-pass over all findings. Replaces unsupported superlatives ("dominates", "overwhelmingly", "almost all", "vast majority") with calibrated language unless a confirming percentage literally appears in the cited query output. Uses _has_majority_evidence() (regex check on evidence text) and _has_any_number() as helpers. Runs inside synthesise() after the LLM call, before the result is returned to the pipeline.

Helper: extract_pitfalls()

extract_pitfalls(playbook_texts: list[str]) → str

Extracts the Pitfalls: block from each playbook and returns a formatted checklist string. Called by both the synthesiser (injected into the synthesis prompt) and the reviewer (injected into each per-finding review prompt), so both stages see the same analytical trap warnings.

Helper: format_prior_context()

format_prior_context(synthesis: SynthesisResult) → str

Produces a compact summary of a completed synthesis (~15 lines): headline finding, per-finding title + confidence + first sentence, and identified gaps. Designed to be injected as context when running a follow-up analysis so the model is aware of what the prior run found.

Not active in the Streamlit pipeline. Prior-context injection was tested and reverted — it made follow-up quality worse in practice. The function remains in synthesiser.py and is used by test_followup_quality.py --with-context for A/B comparison testing only. It is not called from app.py.

reviewer.py Reviewer

Reviews the synthesis against raw executor evidence. Runs one LLM call per finding concurrently, plus deterministic pre-checks. Returns typed flags per finding; correctable flags trigger the revision loop in pipeline.py.

review(
  synthesis: SynthesisResult,
  exec_result: ExecutionResult,
  playbook_texts: list[str],
  model_name: str = PLANNER_MODEL,
  verbose: bool = False
) → ReviewResult

Reviewer Flag Schema

TypeMeaningCorrectableConfidence cap
FALSE_CLAIM Stated X; evidence says Y or is absent Yes→ LOW
UNSUPPORTED_MAGNITUDE Direction right; "dominates/majority" not backed by numbers Yes→ MEDIUM
MISSING_CAVEAT Conclusion sound; material qualification absent. Triggered by: comparing groups without noting size difference; stating statistical significance without reporting effect size; stating a projection or forecast without conditional language or a confidence range Yes→ MEDIUM
PITFALL Known trap triggered: Simpson's paradox, base rate, ecological fallacy, etc. Yes→ LOW
DQ_CONCERN Data quality limits finding (<15 records or >10% null). Not a synthesiser error. No→ MEDIUM
SCOPE_GAP Goal aspect unaddressed; needs new executor queries No—

Overall Assessment Logic

if any(correctable_flags): assessment = "FAIL"   # triggers revision loop
elif any(non_correctable_flags): assessment = "REVIEW"  # acknowledged limitations
else: assessment = "PASS"

Automated Pre-Checks: run_automated_checks()

run_automated_checks(synthesis: SynthesisResult, exec_result: ExecutionResult) → dict

Runs five deterministic checks before the LLM review. Results are stored in ReviewResult.automated_checks and logged per run. A passing number-grounding check on a finding suppresses any LLM FALSE_CLAIM for that finding — deterministic takes precedence.

Check keyWhat it verifies
all_have_confidenceEvery finding has HIGH / MEDIUM / LOW
valid_query_refsAll supporting_queries indices exist in exec_result
all_results_citedEvery succeeded executor query is cited by ≥1 finding
no_orphan_findingsNo finding has empty supporting_queries
number_groundingNumbers in interpretations literally appear in cited query output (reads full untruncated result_text)

Per-Finding LLM Review: _review_single_finding()

The reviewer runs one LLM call per finding, concurrently via ThreadPoolExecutor — the same wave pattern as the executor. Each call receives only that finding's evidence block (full char budget, no truncation pressure from other findings).

# reviewer.py — review() orchestration
failed_block = _format_failed_queries_block(exec_result)
pitfalls     = extract_pitfalls(playbook_texts)

futures = {}
with ThreadPoolExecutor() as pool:
  for idx, finding in enumerate(synthesis.findings):
    fut = pool.submit(_review_single_finding,
                       idx, finding, all_findings,
                       goal, failed_block, pitfalls,
                       exec_result, model_name)
    futures[fut] = idx

# Each call returns: (flag_entry_dict, raw_response, error_str)
# flag_entry: {finding_index, flags: [{type, detail}], reviewer_note}

After all per-finding calls complete, _suppress_false_claims_where_grounded() runs before the overall assessment is derived. It suppresses LLM FALSE_CLAIM flags where the deterministic number-grounding check passed — but only for purely numeric formatting claims. Directional and categorical FALSE_CLAIMs (e.g. "X is the largest category" when Y is actually larger) are preserved from suppression even when the cited number grounds, detected via _is_directional_false_claim() and _DIRECTIONAL_KEYWORDS_RE. _compute_overall_assessment() then derives PASS / REVIEW / FAIL from the merged flag set.

Revision Loop

# pipeline.py — _targeted_resynthesize()
for round in range(config.max_revision_rounds):   # default: 2
  if not _has_correctable_flags(review): break

  # Build evidence block for flagged findings only
  # For FALSE_CLAIM: include ALL query outputs so model can find correct citation
  # For other types: include only currently-cited queries
  flagged_block = _format_flagged_findings_for_revision(synthesis, review, exec_result)

  # Targeted patch: only flagged indices allowed in response
  revised_findings = LLM_patch(flagged_block, model=config.synthesiser_model)

  # Safety guard: discard any index the model returns that wasn't flagged
  synthesis = _merge_revised(synthesis, revised_findings, flagged_indices_only=True)

  review = review(synthesis, exec_result, playbooks)

# Post-loop: cap confidence of findings that still carry flags
synthesis = _apply_confidence_adjustments(synthesis, review)
Supporting query indices are never updated by the revision LLM. supporting_queries and chart_ref are always restored from the original synthesis after a revision response. The model can only change title, interpretation, confidence, and caveats.

AST Code Fixes

Three deterministic post-processing rewrites applied to generated code by ask_agent_xml() before execution. Each targets a specific class of model error that cannot be caught by the executor prompt alone.

FunctionTriggerRewrite
fix_groupby_axis_orientation() .unstack() present; categorical column precedes time column in groupby list Swaps column order: .groupby(['cat', 'month']) → .groupby(['month', 'cat'])
fix_single_index_unstack() .unstack() after single-key groupby (regular Index, not MultiIndex) Removes the .unstack() call entirely
fix_ratio_of_means() df['A'].mean() / df['B'].mean() pattern (ratio of aggregates) Rewrites to (df['A'] / df['B']).mean() (mean of row-level ratios)
All three fixes print a log line when triggered ([post-process] ...) so transformations are visible without verbose mode. They run silently — the model never sees that its code was changed.

Chart Stylesheet & Post-Execution Processing

Applied inside run_sandboxed() — the model's generated code runs inside this wrapper and is unaware of either stage.

Before execution — chart stylesheet

matplotlib.rcParams.update() is called before the executor code runs. The stylesheet:

  • Detects macOS dark/light mode at runtime via defaults read -g AppleInterfaceStyle
  • Sets #FF4B4B (Streamlit red) as the default single-series colour
  • Removes gridlines; sets consistent 11pt font; matches background to system theme

After execution — figure cleanup and time-axis guard

Applied to all axes before fig.savefig():

  • Top and right spines are removed on every axis
  • Time-axis ordering guard: if all x-axis tick labels are 4-digit integers in the 1900–2200 range and are in descending order, ax.invert_xaxis() is called. Catches the common case of year bar charts sorted newest-first by a sort_values(ascending=False) call. Executor Rule 21 instructs the model to sort time data ascending before plotting; this guard is the fallback.

Startup & Diagnostics

check_ollama(model_name: str = MODEL_NAME) → {"ok": bool, "error": str}

Pings OLLAMA_HOST and confirms the named model is in the model list. Returns {"ok": True} on success, or {"ok": False, "error": "..."} with a fix hint. Called at Streamlit startup.

verify_collections(collections: dict, persist_dir: str) → {"ok": bool, ...}

Opens the ChromaDB store and calls get_collection() on each named collection. Guards against post-upgrade schema mismatches that would otherwise surface mid-pipeline. Fix hint: re-run the relevant setup_stores_*.py or re-upload the dataset.

The executor LLM is instructed to respond strictly in XML. Any response not matching this structure is treated as a failed query.

Dataset removal

DataFrame Rendering — Print Interceptor

run_sandboxed() replaces the print builtin in the sandbox's allowed_builtins with _sandboxed_print. This interceptor captures DataFrame and Series objects for rich rendering in Streamlit, while still writing the string representation to the output buffer as normal.

# agent.py — inside run_sandboxed()
captured_frames = []

def _sandboxed_print(*args, sep=" ", end="\n", file=None, flush=False):
  for arg in args:
    if isinstance(arg, pd.DataFrame):
      captured_frames.append(arg.copy())          # stash the object
    elif isinstance(arg, pd.Series):
      captured_frames.append(arg.reset_index())   # promote index to columns
  _original_print(*args, sep=sep, end=end)        # still writes to stdout buffer

allowed_builtins["print"] = _sandboxed_print

# After exec: bare last-expression DataFrames also added to captured_frames
return {"ok": True, "stdout": stdout, "frames": captured_frames, ...}

executor_loop.py propagates frames through the per-query result dict. app.py's _render_stdout(text, frames) renders each captured frame via st.dataframe() — proper string-left / number-right alignment, sortable columns, horizontal scroll. Plain text and uncaptured Series fall back to st.code().

Generated code must use print(result), not print(result.to_string(...)). The interceptor needs the object itself to call st.dataframe() on it. A pre-converted string bypasses the interceptor and falls back to code block rendering. This is Executor Rule 20.

Next-Step Buttons & Result Stacking

After an analysis run, next_steps_now items from the SynthesisResult are rendered as st.button() widgets below the findings. The session state machinery avoids Streamlit's "can't modify widget key after instantiation" constraint.

# app.py — next-step button click handler
if st.button(step_text, key=f"next_{result_idx}_{step_idx}"):
  st.session_state["pending_goal"]  = step_text   # deferred — not set on widget directly
  st.session_state["auto_run"]      = True
  st.rerun()

# app.py — at the top of render_analysis_tab(), before rendering
if st.session_state.get("auto_run"):
  goal = st.session_state.pop("pending_goal")
  st.session_state["auto_run"] = False
  result = run_analysis(goal, ...)
  st.session_state["analysis_results"].append(result)   # stacks below prior results

Result stacking behaviour

  • Next-step runs append to analysis_results — prior results remain visible above
  • Manual "Run Analysis" clears analysis_results — fresh start
  • Each result has its own PDF export button, keyed by list index (idx) to avoid DuplicateWidgetID errors
  • The pipeline execution block runs after the results loop so existing results render before the spinner appears

RAG Architecture

Two ChromaDB stores per dataset, embedded with mxbai-embed-large via Ollama. Retrieval is weighted across both stores; scores are normalised and merged before top-N selection.

Playbook Chunk Schema

Each playbook is stored as a JSONL record. Key fields retrieved and injected into planner/synthesiser/reviewer prompts:

Embed truncation. setup_stores_playbooks.py truncates each playbook's embed text to 1800 characters before calling mxbai-embed-large (context limit). The full text is stored in ChromaDB metadata and returned at retrieval time — only the embedding is truncated.

Logging

Every pipeline run appends one JSONL record to pipeline_logs/pipeline_{dataset}_{date}.jsonl regardless of outcome. The log is written after the final stage completes (or after the first stage that fails). It is the authoritative record of what happened and the primary input for all evaluation, consistency analysis, and HTML report generation.
