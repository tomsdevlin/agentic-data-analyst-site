Agentic AI Analyst

A self-directing analysis mode where the AI decides what to investigate, writes and runs its own code, and writes up findings when it judges it has enough evidence. No fixed plan. No manual steps.

A different kind of analysis

The standard pipeline follows a fixed sequence: a planner breaks the goal into questions, an executor runs them in order, and a synthesiser writes up the results. The agentic pipeline works differently — the model is given a goal and a set of tools, and decides for itself how to investigate it.

Instead of running a pre-determined list of queries, the agentic model explores the data iteratively. It starts an investigation, sees the result, then decides what to look at next — exactly as an analyst would. If an early result reveals something unexpected, it can follow that thread. If a query produces nothing useful, it moves on.

Standard pipeline

Plan, then execute

A planner writes a fixed list of questions upfront. The executor runs them in order. Analysis is structured but follows a predetermined path.

  • Questions fixed before any data is seen
  • Each stage hands off to the next
  • Efficient for well-defined goals
Agentic pipeline

Investigate, then synthesise

The model explores the data iteratively, choosing each next step based on what it just found. It synthesises when it judges it has enough evidence.

  • Adapts as evidence accumulates
  • Can follow unexpected threads
  • Better suited to open-ended goals

The agent loop

The model runs in a loop. Each turn it reviews the evidence so far, chooses a tool to call, observes the result, and decides whether to continue or to synthesise. The loop runs until the model calls synthesise or the turn budget is exhausted.

The tools

The model's understanding of what it can do comes from tool definitions — structured descriptions of each available action. There are five tools. Four are investigation tools; the fifth ends the analysis.

{ }
execute_code
Write and run Python against the dataset

The primary investigation tool. The model writes Python using pandas — filtering, grouping, correlating, running statistical tests — and the code executes in a sandboxed environment against the real dataset. The result (output, any tables, error if it failed) comes back on the next turn.

Common scientific libraries are pre-loaded into the sandbox: NumPy, SciPy, and statsmodels are all available without needing an import. The model can run regressions, hypothesis tests, and distribution checks as naturally as basic aggregations.

A single execute_code call can also generate a chart — the model decides whether a visualisation would be useful for a given query. Charts are also generated in a dedicated post-synthesis charting phase after the analysis is complete.

Input
A Python code string, using df to refer to the loaded dataset
Returns
Stdout output, any printed DataFrames, any chart generated, or an error message
≡
get_schema
Inspect the dataset's column structure and data quality summary

Returns the dataset schema: column names, data types, non-null counts, and a basic quality summary for each field. The schema is also included in the system prompt at the start of each run, so the model typically calls this only to recheck details mid-analysis rather than to orient itself from scratch.

Input
None required
Returns
Column-level schema with types, row counts, and basic completeness information
↗
retrieve_context
Query the data dictionary for column descriptions and field definitions

Runs a semantic search against the dataset's data dictionary — a library of column descriptions, categorical field definitions, and known data quality notes. The model uses this to clarify what a field means before using it in an analysis.

In practice, the plain-text data dictionary is loaded into the system prompt at run start, so the model typically doesn't need to call this tool explicitly. It's available for deeper look-ups when the model wants more detail than the summary in the prompt provides.

Input
A plain-English query, e.g. "what does room_type represent?"
Returns
Ranked chunks from the data dictionary most relevant to the query
📌
record_finding
Anchor key numbers mid-analysis before the context moves on

After completing a significant piece of analysis — computing a trend, identifying a top group, running a statistical test — the model calls record_finding to pin the specific numbers it just found. Each recorded finding is returned on every subsequent turn, so key results stay available throughout the analysis without the model needing to re-run queries it already completed.

This matters because the analysis loop can run for many turns across many dimensions. Without anchoring, exact figures discovered early can drift or be misquoted in the synthesis. The tool's rejection logic enforces that the finding field is populated with real numbers — an empty call is rejected with a directive to copy the result from the immediately preceding output.

Input
A short label (the dimension) and a finding with specific numbers copied from the preceding output
Returns
All findings recorded so far in the current run, returned on every subsequent call
✎
synthesise
Write up findings and close the analysis loop

Called when the model judges it has investigated enough dimensions to address the goal. The model writes a structured summary that draws on its accumulated observations and any anchored findings — explaining what the data shows, with specific figures, and noting where the data was insufficient or ambiguous.

Calling synthesise ends the analysis loop. The synthesis passes directly to the charting phase and then to the reviewer. The model cannot make further code calls after synthesising — observations that weren't captured before this call are not available for review.

Input
All accumulated observations from the analysis loop
Returns
A written synthesis — the structured output that the reviewer checks and the user sees

How a run unfolds

Every agentic run passes through three phases in sequence. The analysis loop is where the investigation happens; the charting phase adds visualisations; the review phase checks the written output for errors.

Phase 1
Up to 25 turns

Analysis loop

The model investigates the goal by calling tools in sequence — running code, recording key numbers, following threads as they emerge. Each turn it sees all prior observations and decides what to do next. When it judges the analysis complete, it calls synthesise to write up the findings and exit the loop.

Phase 2
Up to 7 turns

Charting phase

After synthesising, the model enters a separate charting phase with the synthesis and column schema in context. It generates any charts that would help illustrate the findings — bar charts, time series, distributions — then stops when done. Charts are embedded directly in the HTML report alongside the written output.

Phase 3
1 round

Review & revision

A reviewer model checks the synthesis against the accumulated observations. Claims that can't be traced to the evidence, missing qualifications, or known analytical traps are flagged. If correctable problems are found, a single revision round is triggered — the model rewrites the affected sections. The revised synthesis is then reviewed once more to confirm the issues were resolved.

Quality checks

The reviewer classifies every problem it finds into one of six types. The type determines what happens next: correctable problems trigger a revision round; advisory problems are flagged without blocking the output.

✗ False Claim

The synthesis states a number or direction that the observations contradict. The section is rewritten to reflect what the data actually shows.

✗ Unsupported Magnitude

Words like "dominates" or "the majority" appear without a percentage to back them. The claim is requalified with the actual figures.

✗ Missing Caveat

The conclusion is directionally correct but omits a material qualification — for example, basing a comparison on fewer than 15 records without noting the small sample.

✗ Analytical Pitfall

A known statistical trap has been triggered — for example, reporting a strong correlation between a component and its total (which is mathematically guaranteed, not analytically informative).

⚠ Data Quality Concern

The data limits confidence in the finding — for example, fewer than 15 records in a comparison group, or more than 10% missing values in a key field. Not correctable; flagged for the reader.

⚠ Scope Gap

A dimension of the original goal was not investigated, typically because the data wasn't sufficient to support it. Flagged so the reader knows what was left out.

Flags require evidence, not suspicion. Every correctable flag must include a direct quote from the synthesis, a direct quote from the observations that contradicts it, and a confidence score. Only flags with a confidence of 0.8 or above trigger revision — lower-confidence concerns are recorded and shown in the report but do not cause a rewrite. This prevents a reviewer that is merely uncertain from triggering a revision that makes the synthesis worse.
Confidence is adjusted after review. Findings corrected for a False Claim or Pitfall are capped at Low confidence. Findings with Unsupported Magnitude, Missing Caveat, or Data Quality concerns are capped at Medium. High confidence is reserved for findings that passed review without correction.

Under the hood

Several mechanisms shape how the agentic model runs — some borrowed from the standard pipeline, others specific to the loop-based architecture.

Privacy by design

Your data and the AI models never leave your system

Like the standard pipeline, the agentic mode runs entirely on local hardware. The language models are downloaded and run via Ollama. No data is transmitted to an external API at any point — not the dataset, not the goal text, not the observations. Everything stays on your machine.

Sandboxed execution

Code runs in a restricted environment, not a general Python interpreter

Every execute_code call runs in the same sandbox used by the standard pipeline. The model can only access the specific dataset file; it cannot read other files on your system, make network calls, or write to disk. Each code call is isolated — code from one turn cannot affect the next.

  • Data access. Only the target CSV is accessible. No other files.
  • No network. The sandbox has no outbound network access.
  • Isolated turns. Each call has a fresh execution context; state does not bleed between turns.
  • Blocked patterns. File writes, system calls, and subprocess launches are intercepted before execution.
Loop controls

Turn budget and repeat-loop guard keep runs on track

The analysis loop runs for a maximum of 25 turns. If the model hasn't synthesised by then, the run ends and whatever evidence has been collected is written up.

A repeat-loop guard detects when the model submits identical code three turns in a row — a pattern that indicates it has become stuck retrying a query that isn't working. When this happens, the model receives a nudge to either move on to a different dimension or call synthesise with the evidence it already has.

  • Turn budget. 25 turns for analysis, 7 for charting — set in config, overridable per run.
  • Repeat guard. 3 identical submissions triggers a redirect nudge.
  • No-tool guard. 2 consecutive turns with no tool call triggers an urgent nudge — prevents the model from narrating without acting.
Data dictionary

Column context loaded at the start of every run

The plain-text data dictionary — describing what each column means, what categorical values represent, and how fields relate to each other — is embedded directly into the model's system prompt at run start. The model begins with semantic understanding of the data, not just column names and types.

This is particularly important for the agentic mode because there is no planner stage that might otherwise reframe ambiguous columns. Without dictionary context, the model can misinterpret field names and produce analyses that are technically correct but semantically wrong.

Model freedom vs guardrails

The model reasons freely; critical checks are deterministic

The agentic model is deliberately given more freedom than the standard pipeline's executor — it chooses its own analytical path. But several operations remain deterministic:

  • Confidence gate. Only reviewer flags with confidence ≥ 0.8 trigger a revision round. A flag that the reviewer is uncertain about cannot force a rewrite that might make the synthesis worse.
  • Record finding enforcement. An empty record_finding call is rejected in code, not by the model. The rejection message instructs the model to copy exact numbers from the preceding output and call again immediately.
  • Bounded revision. The revision loop runs a maximum of one round. The model cannot enter an indefinite correction cycle.
  • Chart deduplication. The charting phase tracks code submissions by hash. Identical code submitted twice is blocked immediately — the model cannot repeat the same chart for remaining turns.
